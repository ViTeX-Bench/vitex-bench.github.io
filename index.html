Replacing scene text on signs, jerseys, screens, blackboards, and product labels in real-world videos while preserving the surrounding content, motion, lighting, and camera dynamics.

Abstract

Video scene text editing aims to replace text appearing on signs, whiteboards, jerseys, screens, product labels, and other scene surfaces in a video while preserving the surrounding content, motion, lighting, and camera dynamics. Although scene text editing has been extensively studied for still images, its video counterpart remains underdeveloped: existing resources lack paired real-video edits, current evaluation protocols do not measure character-level correctness over time, and no open reference model exists for systematic comparison. We introduce ViTeX-Bench, a benchmark suite for faithful and temporally consistent video scene text editing. Specifically, we build ViTeX-Dataset, which contains 387 curated real-world 720p video clips, sourced from Panda-70M and InternVid. We construct it through a semi-automatic pipeline that localizes scene text, creates source–target text pairs, removes the original text, and synthesizes temporally consistent edits. We then establish ViTeX-Bench, which evaluates video scene text editing models along three axes: text correctness, visual quality, and edit locality, using thirteen complementary metrics aggregated into a single ViTeX-Score. To demonstrate the effectiveness of our paired dataset, we further introduce ViTeX-14B, an open model based on Wan2.1, with a glyph-video conditioning pathway that supplies temporally aligned target-text structure to the video-editing backbone. Together, ViTeX-Dataset, ViTeX-Bench, and ViTeX-14B provide a common foundation for training, evaluating, and diagnosing video scene text editing systems.

Each clip ships with the source video, a propagated per-frame binary mask of the editable text region, and a source–target string pair. The training split additionally provides a paired edited video produced by our two-strategy construction pipeline (alpha composition for static-text clips, a fine-tuned PISCO inserter for dynamic clips). The 157-clip evaluation split is permanently frozen and withholds edited ground truth, so the benchmark cannot leak edit-side information into training. Source and target strings range from single characters to ~40-character phrases, covering both short edits (where per-frame editors can in principle recover the full string) and longer phrases that stress temporal consistency.

Evaluation protocol

Three orthogonal axes, thirteen complementary metrics, one non-substitutive aggregate.

Axis 1

Text correctness

PP-OCRv5 with per-clip language routing reads the dilated text-region crop; source-detectability gating restricts scoring to frames where the source is readable. Substring (semi-global) edit distance prevents penalizing a correct target embedded in a longer OCR string.

  • SeqAcc — exact-match rate
  • CharAcc — substring similarity
  • TTS — temporal text stability
Axis 2

Visual quality

Six primitives at two scopes (full frame and a static union-mask text crop). Flicker rewards smoothness (hackable by interpolation); Warp uses RAFT optical flow on the source video and only scores low when the prediction follows the same per-pixel motion.

  • Flicker_f / Flicker_c — adjacent-frame MAE
  • Warp_f / Warp_c — RAFT-flow warping error
  • MUSIQ_f / MUSIQ_c — KonIQ-pretrained no-reference IQA
Axis 3

Edit locality

Compose a locality-only prediction by copying source pixels into the masked region, then score four metrics against the source. DreamSim is robust to the codec/VAE noise that pixel-level metrics over-penalize on full-frame regenerative methods.

  • PSNR_loc / SSIM_loc
  • LPIPS_loc (AlexNet)
  • DreamSim_loc — learned identity
ViTeX-Score = ∛(Correctness · Visual · Locality)
Axis = geomeanw(m̃1, …, m̃k),    m̃i = normalize(mi; frozen v1 endpoints) ∈ [0, 1]
Geometric mean at both levels enforces non-substitutivity: any axis at zero collapses ViTeX-Score, so visual quality cannot rescue a missing edit. Inside the visual axis the text-crop scope carries double weight (wcrop = 2, wfull = 1); correctness and locality use uniform weights.

Why three axes, jointly

Four representative baseline failures, each isolated by a different axis or pair. A single visual impression or a single metric column would miss them.

A
High TTS ≠ successful edit
video placeholder

A method that consistently preserves the source string registers high TTS but SeqAcc = 0 (Wan2.1-VACE-14B in zero-shot mode). Reading TTS jointly with SeqAcc disambiguates "consistently rendering the target" from "consistently doing nothing".

B
High visual quality ≠ correctness
video placeholder

Kling Video 3.0 Omni leads both MUSIQ axes with SeqAcc = 0: visually plausible glyphs that spell the wrong target. A benchmark aggregating visual metrics without conditioning on correctness would rank this first.

C
Per-frame ≠ temporally stable
video placeholder

FLUX-Text reaches strong SeqAcc and CharAcc on individual frames yet exhibits glyph-identity drift across the video — high per-frame correctness co-occurs with low TTS and an extreme Flicker_c spike.

D
Propagation collapses correctness and locality
video placeholder

TextCtrl + AnyV2V begins with a correct first-frame edit, then accumulates off-mask drift over time — locality metrics (PSNR / SSIM / LPIPSloc) collapse together with text correctness as propagation progresses.
